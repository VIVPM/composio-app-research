The agent — what we built

A two-layer research pipeline

Cheapest-source-first: let Composio's catalog answer what it can objectively, and only spend an LLM on what it can't. Every field carries provenance and a confidence.

LAYER 1 · GROUND TRUTH

Composio catalog

composio_lookup.py pulls all 1,047 toolkits from the v3 REST API and matches each app. For the 55 it ships, auth scheme, tool count and buildability come straight from the catalog — no guessing.

LAYER 2 · WEB + LLM

Doc extraction

extract.py (pluggable LLM, web-search grounded) fills description, self-serve vs gated, REST/GraphQL, first-party MCP, blocker and evidence — for all 100, and is the sole source for the 45 Composio doesn't have.

MERGE · PROVENANCE

Orchestrator

research.py merges the layers into results.json: Composio wins on auth & breadth where present; the doc layer fills the rest; each record keeps its source + confidence.

Where a human was needed (stated honestly)

  • Fuzzy-match false positives. The matcher wrongly linked Zoho Cliq→zoho, Squarespace→square, Plaid→placid, SF Commerce Cloud→salesforce. A human rejected all four.
  • Under-detected MCPs. The first pass marked several apps as having no first-party MCP; browser verification caught Google Ads and SE Ranking (both ship one). This field is still the least certain.
  • The niche long tail. fanbasis, iPayX, Paygent, Waterfall.io have thin/absent public docs — "gated / no clear public API" is the honest finding, not a confident spec.
  • One doc was unreachable. Otter.ai's MCP page returned HTTP 403 — held at first-pass, flagged.
The proof — it runs

A working Composio agent on an easy win

GitHub scored top-right on the map: self-serve OAuth/PAT, REST+GraphQL, 871 Composio tools. So the repo ships a tiny runnable agent that uses Composio to act on GitHub.

Runnable trigger

Connects a GitHub account via Composio and creates/searches an issue. Full source in proof/github_agent.py.
export COMPOSIO_API_KEY=...        # same key the research agent uses
python proof/github_agent.py "Ship the case study" --repo you/yourrepo
# -> authorizes GitHub (once), then creates the issue and prints its URL
The verification — how we know it's trustworthy

Two loops moved accuracy up

Loop 1 — auth cross-check · objective, 0 web calls

The first pass guessed auth independently. We scored those guesses against Composio's catalog ground truth on the 55 apps it ships.

96%matched the real auth (≥1 shared method)
62%exact set match — the gap is us over-listing methods, not wrong ones

Only 2 true misses: Telegram & Vercel — both "token" vs "API key", a semantic hair.

Loop 2 — browser/web sample · 20 apps, 2 per category

A human opened the live docs for a stratified sample and scored self-serve/gated, auth and MCP.

96%→100% first pass → after correcting the misses

Both misses were the same field — a missed first-party MCP. Correcting them lifted the overall MCP count 28 → 30.
